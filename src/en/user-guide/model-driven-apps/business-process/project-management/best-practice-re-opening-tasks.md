# Best practice: Re-opening tasks

The Loop guidance describes two groups of stakeholders that often see tasks differently:

- people focused on the process, where tasks are side effects of the process moving forward
- people focused on the work itself, where completing the task is the main indicator of progress

If a finished task needs to happen again because the process was rolled back, those groups can disagree about whether the original task should be re-opened.

## Why re-opening is problematic

This becomes especially sensitive when tasks feed KPI reporting or individual performance reporting. Re-opening the original task can make it look as if people took too long, even when they finished the task on time and the process changed later.

## Recommended practices

- Users decide when a task is closed. Responsible people update progress, while accountable people confirm the final status.
- Rollbacks do **not** re-open existing tasks. Instead, they create a new instance of the same task.
- The new instance keeps the same reporting identity so reporting can still group the related work correctly.
- Additional instances should be marked as **corrections** either by tags or by links to cases, depending on company needs.
- If corrections need to be shown separately, a case-based approach is more useful.
- Reports should use the last instance of a task to determine process timing.

## Practical note

Creating those new instances is expected to use partial templates. Users should also keep dependencies accurate so automatic scheduling continues to reflect reality.
